# Exercise 5 — Third-Party Dependency Security Audit

## Objective

This exercise analyzes a vulnerable Node.js `package.json`, scans third-party dependencies for known vulnerabilities, documents high and critical findings, explains how to remediate them, and provides security recommendations.

The exercise also includes a bonus Python script that parses `npm audit --json` output and generates a CSV report containing high and critical findings.

## Files

| File | Purpose |
|---|---|
| `package.json` | Vulnerable Node.js dependency manifest provided by the assessment |
| `package-lock.json` | Lockfile generated so `npm audit` can resolve the full dependency tree |
| `audit_to_csv.py` | Python script that runs or parses `npm audit --json` and generates a CSV report |
| `reports/npm-audit.json` | Raw `npm audit --json` output |
| `reports/npm-audit-high-critical.csv` | CSV report with high and critical audit rows |

## Tool Used

This exercise uses `npm audit`.

`npm audit` analyzes the dependency tree and reports known vulnerabilities affecting direct and transitive dependencies. The Python script processes the JSON output and extracts high and critical findings into a CSV report.

## How to Run

From this folder:

    exercises/05-third-party-dependency-audit

Install dependency metadata and generate the lockfile:

    npm install --package-lock-only --ignore-scripts

On Windows PowerShell, if `npm` is blocked by execution policy, use:

    npm.cmd install --package-lock-only --ignore-scripts

Run `npm audit` manually and save the JSON report:

    npm audit --json > reports/npm-audit.json

On Windows PowerShell, use:

    npm.cmd audit --json | Set-Content -Encoding utf8 reports/npm-audit.json

Then generate the CSV report:

    python audit_to_csv.py --input reports/npm-audit.json --output reports/npm-audit-high-critical.csv

Alternatively, the Python script can run `npm audit --json` directly:

    python audit_to_csv.py

The script is Windows-aware and uses `npm.cmd` automatically when running on Windows.

## Finding Count Interpretation

The 45 high/critical entries are CSV report rows generated by the parser. They should not be interpreted as 45 unique CVEs, 45 unique advisories, or 45 separate remediation actions.

`npm audit` may report:

- Multiple advisories affecting the same package.
- One vulnerable transitive dependency appearing through multiple dependency paths.
- Multiple vulnerable packages fixed by upgrading a single parent dependency.
- A single remediation action that resolves several findings.
- Some fixes that require semver-major upgrades and therefore additional compatibility testing.

For remediation planning, I would distinguish between the following metrics:

| Metric | Why it matters |
|---|---|
| High/Critical CSV rows | Shows the total reporting volume from the parser |
| Unique vulnerable packages | Shows what packages likely need upgrade, removal, or replacement |
| Unique advisories / CVEs / GHSAs | Shows the distinct risk items |
| Dependency paths | Shows how vulnerable packages enter the dependency tree |
| Fix actions | Shows how many actual package changes may be required |
| Semver-major fixes | Shows which remediations may introduce breaking changes |

The CSV is useful for vulnerability review and traceability, but remediation planning should be based on deduplicated packages, advisories, dependency paths, and fix actions.

## Risk-Based Remediation Plan

The preferred remediation order is based on runtime impact and exploitability, not only on severity labels.

| Priority | Category | Rationale | Action |
|---|---|---|---|
| 1 | Critical vulnerabilities in runtime dependencies | These affect the production attack surface directly | Upgrade or replace the affected direct/parent dependency first |
| 2 | High vulnerabilities in runtime dependencies | These may still affect production exposure | Upgrade to the minimum patched version or latest compatible version |
| 3 | Vulnerable transitive dependencies | These may be fixed by upgrading parent packages | Upgrade the parent dependency that introduces the vulnerable package |
| 4 | Dev-only vulnerabilities | Lower production exposure, but still relevant for CI/developer environments | Upgrade dev dependencies such as `mocha` |
| 5 | Semver-major fixes | May introduce breaking changes | Review changelogs, test compatibility, and apply in a controlled branch |
| 6 | Unused or abandoned dependencies | Reduces attack surface and maintenance burden | Remove or replace the dependency |

## How to Fix the Issues

The remediation approach depends on whether the vulnerable package is a direct dependency, a transitive dependency, a dev dependency, or part of an outdated dependency chain.

### 1. Upgrade direct vulnerable runtime dependencies

Direct runtime dependencies should be fixed first because they are more likely to affect the production application.

| Dependency | Current Version | Safer remediation approach |
|---|---:|---|
| `lodash` | `4.17.10` | Upgrade to at least `4.17.21`, or a later compatible patched version |
| `express` | `4.15.0` | Upgrade to a current supported Express version; review release notes and test routing/middleware behavior |
| `body-parser` | `^1.19.0` | Upgrade to a patched `1.20.x` version or later compatible version |
| `jsonwebtoken` | `8.1.0` | Upgrade to `9.x` or later; review breaking JWT signing/verification behavior before production rollout |
| `mongoose` | `^4.2.4` | Plan a staged migration to a supported Mongoose major version; this may require code changes |
| `mocha` | `5.0.5` | Upgrade to a current supported Mocha version; prioritize after runtime dependencies |

Recommended lab upgrade direction:

    npm install lodash@latest
    npm install express@latest
    npm install body-parser@latest
    npm install jsonwebtoken@latest
    npm install mongoose@latest
    npm install --save-dev mocha@latest

On Windows PowerShell, use `npm.cmd` if `npm` is blocked by execution policy:

    npm.cmd install lodash@latest
    npm.cmd install express@latest
    npm.cmd install body-parser@latest
    npm.cmd install jsonwebtoken@latest
    npm.cmd install mongoose@latest
    npm.cmd install --save-dev mocha@latest

### 2. Prefer minimum patched versions when compatibility risk is high

Using `package@latest` is acceptable for a lab exercise, but in production I would first identify the minimum patched version that resolves the vulnerability.

| Situation | Preferred approach |
|---|---|
| Patch/minor upgrade fixes the vulnerability | Upgrade to the minimum patched version |
| Only a major version fixes the vulnerability | Review breaking changes and test in a dedicated branch |
| Package is unsupported or abandoned | Replace the dependency |
| Dependency is unused | Remove it |

### 3. Fix transitive vulnerabilities through parent upgrades

If the vulnerable package is transitive, I would avoid manually editing nested dependencies. Instead, I would identify the parent package that introduces it and upgrade that parent.

Examples:

- If a vulnerable package enters through `express`, upgrade `express`.
- If a vulnerable package enters through `mongoose`, upgrade `mongoose`.
- If a vulnerable package enters through `mocha`, upgrade `mocha`.

After upgrading parent packages, regenerate the lockfile and rerun the audit.

### 4. Use `npm audit fix` for safe automatic fixes

First try non-breaking automatic fixes:

    npm audit fix

On Windows PowerShell:

    npm.cmd audit fix

Then rerun:

    npm audit

and regenerate the CSV report:

    python audit_to_csv.py

### 5. Use `npm audit fix --force` only after review

Some fixes require major version upgrades. These may introduce breaking changes.

Use this only after reviewing release notes and running tests:

    npm audit fix --force

On Windows PowerShell:

    npm.cmd audit fix --force

This is especially relevant for old packages such as `mongoose@4.x`, `jsonwebtoken@8.x`, and `mocha@5.x`, where remediation may require a major version upgrade.

### 6. Replace dependencies when upgrading is not enough

If a vulnerable package has no patched version, is no longer maintained, or cannot be upgraded safely, the correct remediation may be replacement.

Examples of replacement decisions:

| Situation | Fix |
|---|---|
| Package has no patched version | Replace the dependency |
| Package is abandoned or no longer maintained | Replace with a supported alternative |
| Upgrade requires large application changes | Plan a controlled migration |
| Dependency is unused | Remove it from `package.json` |
| Transitive dependency remains vulnerable after parent upgrade | Replace or remove the parent package |

Example commands:

    npm uninstall vulnerable-package
    npm install safer-maintained-alternative

### 7. Remove unused dependencies

If any vulnerable dependency is not actually required by the application, the best fix is removal.

Example:

    npm uninstall unused-package

This reduces both vulnerability exposure and dependency maintenance burden.

### 8. Validate after remediation

After upgrading, replacing, or removing dependencies:

    npm test
    npm audit
    python audit_to_csv.py

The target state should be:

- No critical vulnerabilities.
- No high vulnerabilities unless formally accepted as risk.
- Updated `package.json`.
- Updated `package-lock.json`.
- Application tests passing.
- Any remaining exceptions documented with owner, justification, compensating controls, and expiration date.

## Recommended Remediation Order

I would remediate in this order:

1. Run `npm audit fix` to apply safe non-breaking fixes.
2. Rerun `npm audit` and regenerate the CSV report.
3. Review remaining high and critical findings.
4. Prioritize critical runtime dependencies.
5. Upgrade vulnerable direct runtime dependencies.
6. Upgrade parent packages responsible for vulnerable transitive dependencies.
7. Upgrade dev-only dependencies.
8. Replace abandoned or unfixable dependencies.
9. Use `npm audit fix --force` only after reviewing breaking changes.
10. Run application tests.
11. Commit the updated `package.json` and `package-lock.json`.

## Security Recommendations

To prevent similar dependency risks:

| Control | Purpose |
|---|---|
| `npm audit --audit-level=high` in CI | Fail builds when high or critical vulnerabilities are introduced |
| Dependabot or Renovate | Automatically propose dependency updates |
| Lockfile review | Ensure transitive dependency changes are visible |
| Pin and update dependencies regularly | Reduce exposure to stale vulnerable packages |
| Avoid unsupported major versions | Old frameworks often accumulate unresolved transitive risk |
| Runtime/dev dependency separation | Prioritize production-exposed dependency risk first |
| Require tests before major upgrades | Prevent breaking changes from being deployed blindly |
| Track exceptions | Document owner, justification, compensating controls, and expiration date |

## CI/CD Recommendation

A simple CI process for this project would:

1. Install dependencies using the lockfile.
2. Run `npm audit --json`.
3. Parse the audit output.
4. Fail the build if high or critical vulnerabilities are present.
5. Allow exceptions only when documented and time-bound.

Example policy:

    npm audit --audit-level=high

## Bonus Script

The bonus Python script supports two modes.

Run `npm audit` automatically and generate reports:

Parse an existing audit file:

    python audit_to_csv.py --input reports/npm-audit.json --output reports/npm-audit-high-critical.csv

The script handles the fact that `npm audit` may return a non-zero exit code when vulnerabilities are found. That behavior is expected and does not mean the script failed.

## Notes

This exercise focuses on security analysis and reporting. The vulnerable dependencies are intentionally kept in `package.json` so the audit can reproduce findings.

In a real project, the remediation would be applied, tested, and committed together with the updated `package-lock.json`.
